# 部署与运维记录

## 基础环境

系统：

```text
Ubuntu focal
内存约 2 GB
根分区约 5.9 GB
/userdata 约 8.2 GB
```

建议把服务文件、应用和用户数据放到 `/userdata`，避免根分区空间不足。

## 常用服务

| 服务 | 作用 | 检查命令 |
|---|---|---|
| `nginx` | Web 入口 | `sudo systemctl status nginx --no-pager` |
| `filemgr` | Flask 后端 | `sudo systemctl status filemgr --no-pager` |
| `cloudflared` | 公网隧道 | `sudo systemctl status cloudflared --no-pager` |
| `eth0-direct` | 有线直连 IP | `sudo systemctl status eth0-direct --no-pager` |

## 开机自启检查

```bash
sudo systemctl is-enabled nginx
sudo systemctl is-enabled filemgr
sudo systemctl is-enabled cloudflared
sudo systemctl is-enabled eth0-direct
```

## 服务重启

```bash
sudo systemctl restart nginx
sudo systemctl restart filemgr
sudo systemctl restart cloudflared
sudo systemctl restart eth0-direct
```

## 查看日志

```bash
sudo journalctl -u filemgr -n 80 --no-pager
sudo journalctl -u cloudflared -n 80 --no-pager
sudo journalctl -u nginx -n 80 --no-pager
```


## eth0 直连开机配置

服务文件：

```bash
sudo tee /etc/systemd/system/eth0-direct.service >/dev/null <<'EOF'
[Unit]
Description=Configure eth0 direct link for workstation WOL
After=network.target
Wants=network.target

[Service]
Type=oneshot
ExecStart=/sbin/ip link set eth0 up
ExecStart=/sbin/ip addr flush dev eth0
ExecStart=/sbin/ip addr add 192.168.50.1/24 dev eth0
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target
EOF
```

启用：

```bash
sudo systemctl daemon-reload
sudo systemctl enable eth0-direct
sudo systemctl start eth0-direct
```


### Windows 控制账号

当前用于开发板控制笔记本的 Windows SSH 账号：

```text
用户名：woladmin
用途：远程关机、远程执行应用启动或脚本
连接地址：192.168.50.2:22
```

密码不建议写入笔记。原因：

- 该笔记库会通过 Syncthing 同步到多台设备。
- 账号可以通过 SSH 控制笔记本。
- 明文密码一旦同步泄露，风险较高。

密码实际配置位置：

```text
服务器网页 -> 设备控制 -> SSH 密码
```

后端保存文件：

```text
/userdata/server/apps/filemgr/devices.json
```

如果忘记或修改了 `woladmin` 密码，需要同步更新网页里的 SSH 密码。

Windows 修改密码示例：

```powershell
net user woladmin 新密码
```

然后在服务器网页里重新保存设备配置。

## 故障判断表

| 现象 | 可能原因 | 检查方向 |
|---|---|---|
| 网页打不开 | Nginx 未启动或 Cloudflare 异常 | `systemctl status nginx cloudflared` |
| API 报错 | Flask 后端异常 | `journalctl -u filemgr` |
| 上传失败 | 权限、路径、文件大小限制 | 查看 Flask 日志 |
| 下载慢 | 公网链路限制或 Cloudflare 中转慢 | 局域网和公网分别测速 |
| SSH 离线 | 笔记本关机、SSH 未启动、防火墙拦截 | `ssh woladmin@192.168.50.2` |
| 不能关机 | SSH 登录失败或 Windows 权限不足 | 检查 `woladmin` 权限 |
| 不能开机 | WOL 参数、BIOS、网卡电源管理问题 | 测试 `wakeonlan` |
| 重启后有线不通 | eth0 IP 丢失 | 检查 `eth0-direct` |

## Windows SSH 配置模板

Windows 侧建议固定维护 `C:\Users\20634\.ssh\config`，把开发板的有线、WiFi 两种入口都写进去。

推荐模板：

```sshconfig
# TaishanPi 开发板：网线直连，推荐 VS Code 开发使用
Host taishanpi
    HostName 192.168.50.1
    User lckfb
    Port 22
    IdentityFile C:\Users\20634\.ssh\id_ed25519

# TaishanPi 开发板：当前 WiFi 地址
Host taishanpi-wifi
    HostName 192.168.1.14
    User lckfb
    Port 22
    IdentityFile C:\Users\20634\.ssh\id_ed25519

# Windows 笔记本：供开发板远程关机/执行命令使用
Host jj-windows-lan
    HostName 192.168.50.2
    User woladmin
    Port 22
```

日常建议：

- VS Code 开发优先连接 `taishanpi`
- 只走无线时再连接 `taishanpi-wifi`
- `IdentityFile` 继续使用现有的 `id_ed25519`

测试命令：

```powershell
ssh taishanpi
ssh taishanpi-wifi
```

## 换 WiFi 后恢复 VS Code SSH

开发板更换 WiFi 后，通常是 `wlan0` 的 IP 发生变化，导致 VS Code 还在连接旧地址。

### 1. 在开发板上查看当前地址

```bash
ip a show wlan0
ip a show eth0
hostname -I
```

当前项目中常见两种地址：

```text
WiFi: 192.168.1.14
有线直连: 192.168.50.1
```

### 2. 判断 VS Code 应该连哪个地址

- 电脑和开发板网线直连时：连接 `taishanpi`，即 `192.168.50.1`
- 电脑和开发板同处新 WiFi 时：连接 `taishanpi-wifi`，即 `192.168.1.14`

### 3. 先在 Windows 终端验证

```powershell
ssh taishanpi
ssh taishanpi-wifi
```

如果命令行能连，VS Code 也就可以连。

### 4. VS Code 操作路径

```text
Ctrl+Shift+P
Remote-SSH: Connect to Host
选择 taishanpi 或 taishanpi-wifi
```

### 5. 如果 SSH 仍不通

先在开发板确认 SSH 服务：

```bash
sudo systemctl status ssh --no-pager
ss -ltnp | grep :22
```

如果服务异常：

```bash
sudo systemctl restart ssh
sudo systemctl enable ssh
```

### 6. 固定结论

- 开发板有线地址固定为 `192.168.50.1`
- Windows 有线地址固定为 `192.168.50.2`
- WiFi 变化只影响 `taishanpi-wifi`
- 日常开发优先使用有线别名 `taishanpi`
